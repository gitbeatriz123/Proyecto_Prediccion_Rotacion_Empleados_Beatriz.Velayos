Este informe resume el objetivo de negocio, los datos y su calidad, la arquitectura, el ETL, el EDA, el modelado tradicional y con DL, la comparacin de resultados, los KPIs para negocio y las conclusiones con prximos pasos.

1. Resumen ejecutivo

  • Objetivo: Predecir Attrition (abandono) de empleados para priorizar acciones de retencin.
  • Resultado: Se evaluaron modelos ML (LogReg, RandomForest) y un MLP (DL). Se seleccion el modelo ptimo con umbral operativo para maximizar F1/recall.
  • Impacto: Identificacin de factores clave (ej. OverTime, Income, engagement). KPI de negocio con tasas por JobRole, OverTime, bandas salariales y antigedad.

2. Definicin del problema

  • Tipo: Clasificacin binaria (Attrition: S/No).
  • Mtrica principal: ROC-AUC + F1 (por desbalanceo ~16% de clase positiva).
  • Uso en negocio: Score de salida y bandera (segn umbral) para campaas de retencin, priorizacin de manager HR y poltica de horas extra.

3. Datos y calidad

  • Fuente 1: WA_Fn-UseC_-HR-Employee-Attrition.csv (RRHH).
  • Fuente 2: encuesta_clima.csv (clima laboral; sinttica/real).
  • Clave de unin: EmployeeNumber.
  • Validacin: Nulos y duplicados en clave; fraccin de nulos por columna; status: PASS.

4. Arquitectura (resumen)

  • Docker Compose: Jupyter + Spark (master/worker) + (opcional) PostgreSQL.
  • Ingesta: Volmenes data/ y output/ montados.
  • Procesado: PySpark para ETL y mtricas; scikit-learn/TensorFlow para modelos.
  • Visualizacin: Notebook + exportables (CSV/PNG) y/o Power BI.

5. ETL (Spark) y feature engineering

  • Join inner por EmployeeNumber.
  • Limpieza y normalizacin de strings.
  • Features creadas:
    • attrition_label (Yes1, No0)
    • overtime_flag (Yes1, No0)
    • income_yearly = MonthlyIncome * 12
    • tenure_ratio = YearsInCurrentRole / (TotalWorkingYears + eps)
  • Salida: Parquet en /data/processed/employee_attrition.parquet.

6. EDA (resumen)

  • Clase positiva ~16% desbalanceo moderado.
  • OverTime=Yes y ingresos bajos asocian mayor attrition.
  • Satisfaccin/engagement menores elevan la tasa.

7. Modelado y comparacin

  • ML tradicional: Logistic Regression (balancing), Random Forest.
  • DL: MLP (scikit-learn o Keras).
  • Mtricas: Accuracy, F1, ROC-AUC, PR-AUC.

Se selecciona el mejor modelo y se ajusta umbral operativo (F1/recall segn negocio).

# Cargar mtricas y umbrales guardados
import json, os, pandas as pd

def load_json(p):
    if os.path.isfile(p):
        with open(p) as f: 
            return json.load(f)
    return {}

baseline = load_json("/output/metrics/model_baseline.json")
compare  = load_json("/output/metrics/model_compare.json") or baseline
thr_lr   = load_json("/output/metrics/threshold_tuning.json")
thr_mlp  = load_json("/output/metrics/threshold_tuning_mlp.json")

print("Baseline:", baseline if baseline else "(no encontrado)")
print("\nComparativa:", compare if compare else "(no encontrado)")
print("\nUmbral LR:", thr_lr if thr_lr else "(no encontrado)")
print("Umbral MLP:", thr_mlp if thr_mlp else "(no encontrado)")

# Tabla resumen si existe comparativa
if compare:
    rows = []
    for m, metrics in compare.items():
        rows.append({"model": m, **metrics})
    display(pd.DataFrame(rows).sort_values("roc_auc", ascending=False))

7.1 Poltica de decisin (umbral)

  • Umbral seleccionado por mximo F1 (o por recall objetivo si negocio prioriza capturar ms casos).
  • Se recomienda monitorizar F1/precision/recall mensual y recalibrar.

Salida de inferencia: score [0,1] + prediccin binaria segn umbral.
Ejecucin sugerida: lote semanal (o diaria) y aviso a managers de equipos con mayor tasa prevista.

8. KPIs y recomendacin de negocio

  • Tasa por JobRole y OverTime: priorizar revisin de horas extra y planes de compensacin.
  • Bandas income y tenure: disear polticas de retencin especficas por tramo.
  • Usar la bandera de riesgo del modelo para priorizar entrevistas stay-interview.

9. Conclusiones y lessons learned

  • Modelos ML tradicionales pueden rendir tan bien o mejor que DL con buen preprocesado.
  • La calidad de datos y el umbral impactan ms que micro-ajustes en hiperparmetros.
  • Mtricas deben leerse junto al coste/beneficio (falsos positivos vs falsos negativos).
